PK解析をCDISC標準で効率化！SDTM・ADaM・SASワークフロー徹底解説

Photo from Unsplash

この記事でわかること

  • PK解析（薬物動態）におけるCDISC (Clinical Data Interchange Standards Consortium) 標準の重要性とワークフローがわかります。
  • SDTM (Study Data Tabulation Model) からADaM (Analysis Data Model) へのPKデータマッピングの具体的な手順とSASコード例を学べます。
  • FDA (米国食品医薬品局) やPMDA (医薬品医療機器総合機構) の最新規制要件と、それに対応するためのポイントを把握できます。
  • CRO (医薬品開発業務受託機関) や製薬企業で役立つ、効率的なPK ADaM作成とバリデーションのヒントが得られます。

結論から言うと

PK解析におけるCDISC標準への準拠は、規制当局への申請を円滑に進める上で不可欠です。特にSDTMのPP (関連事象) /PC (薬物濃度) ドメインから、ADaMのBDS (Basic Data Structure) を用いてAUC (薬物濃度時間曲線下面積) やCmax (最高血中濃度) などのPKパラメータを作成するワークフローを確立することが重要です。このプロセスをSASで自動化し、適切にバリデーションすることで、データの品質とトレーサビリティを確保し、FDA/PMDA双方の要件を満たすことができます。

PK解析（薬物動態）におけるCDISC標準ワークフローとは？

PK解析（薬物動態解析）は、薬剤が体内でどのように吸収され、分布し、代謝され、排泄されるかを評価する重要なプロセスです。この解析結果は、薬剤の安全性と有効性を判断し、適切な投与量や投与方法を決定するために不可欠です。

近年、世界中の規制当局（FDA、PMDAなど）は、臨床試験データの提出にCDISC標準の利用を義務付けています。これにより、データの一貫性と品質が向上し、審査プロセスが効率化されることが期待されます。PK解析データも例外ではなく、CDISC標準に沿ってSDTM (Standard Data Tabulation Model) およびADaM (Analysis Data Model) フォーマットで提出する必要があります。

CDISCにおけるPKデータの流れ：

  1. Raw Data (生データ): 臨床試験で測定された個々の薬物濃度データなど。
  2. SDTM (Standard Data Tabulation Model): Raw Dataを標準化された形式（タブレーションデータ）に変換します。PKデータは主にPC (薬物濃度) ドメインとPP (関連事象) ドメインで表現されます。
  3. ADaM (Analysis Data Model): SDTMデータを基に、統計解析に直接使用できる形式（解析データセット）に変換します。PK解析では、AUC (薬物濃度時間曲線下面積) やCmax (最高血中濃度) といったPKパラメータを算出し、これをADaMデータセットとして構築します。
  4. Define-XML (メタデータ): SDTMおよびADaMデータセットの構造、変数、ロジックなどを定義したメタデータファイルです。データセットと同様に提出が求められます。

この一連のワークフローを適切に構築することで、CRO (医薬品開発業務受託機関) や製薬企業は、データの透明性を高め、バリデーションを容易にし、規制当局の要件を満たした高品質なデータパッケージを提供できるようになります。

実務での使い方・実装手順：PKデータとADaM設計

PKデータのADaM設計は、SDTMデータを基に解析用の変数を派生させ、統計解析ツールで扱いやすい形式に整理するプロセスです。ここでは、具体的なSASコード例を交えながら、その手順を解説します。

Step 1: SDTMマッピングの基本（PC/PPドメイン）

PK解析に必要なSDTMデータは主に以下の2つのドメインです。

  • PCドメイン (Pharmacokinetics Concentrations): 各被験者の各時点における薬物濃度測定値とその関連情報。
    • USUBJID (被験者識別子), PCORRES (薬物濃度測定値), PCSTAT (測定ステータス), PCTPTNUM (時点番号), PCTPTREF (時点の参照情報), PCDTC (測定日時) などが重要です。
  • PPドメイン (Pharmacokinetics Parameters): 非コンパートメント解析（NCA: Non-Compartmental Analysis）などで算出された個々のPKパラメータ（AUC, Cmaxなど）をSDTM形式で保持します。
    • USUBJID, PPPARM (パラメータ名), PPORRES (パラメータ測定値), PPSTRESN (標準化された数値結果) など。
    • 補足: PPドメインは、解析前のNCA結果を格納する場所であり、これからADaMで算出するPKパラメータとは少し役割が異なります。ADaMでは解析で使われる最終的なPKパラメータを格納します。

Step 2: PK ADaM設計の考え方（BDSの利用）

PK解析用ADaMは、主にBDS (Basic Data Structure) を用いて構築されます。BDSは、解析対象のデータと解析用の派生変数を1レコードで管理するのに適しています。

ADaMの主要な派生変数ロジック:

  • 主要なPKパラメータ:
    • AUC (Area Under the Curve, 薬物濃度時間曲線下面積): AUCLAST (最終時点までのAUC), AUCINF_OBS (最終観測点から無限大までのAUC) など。
    • CMAX (Maximum Observed Concentration, 最高血中濃度)
    • TMAX (Time of CMAX, 最高血中濃度到達時間)
    • LAMZ (Terminal Elimination Rate Constant, 終末相消失速度定数)
    • HL (Half-Life, 半減期)
    • これらはNCAツール（WinNonlin, Phoenix WinNonlinなど）で計算されることが多く、その結果をADaMに組み込みます。
  • 解析対象フラグ:
    • ANLFL (Analysis Flag): 解析対象となるレコードを示すフラグ（'Y'/'N'）。
    • SAFFL (Safety Analysis Flag): 安全性解析対象レコードを示すフラグ（'Y'/'N'）。
    • EFFICFL (Efficacy Analysis Flag): 有効性解析対象レコードを示すフラグ（'Y'/'N'）。
    • PKFL (Pharmacokinetics Analysis Flag): PK解析対象レコードを示すフラグ（'Y'/'N'）。
  • 時間関連変数:
    • ADTM (Analysis Date/Time): 解析時点の日時。
    • ADY (Analysis Day): 解析日 (ベースラインからの相対日)。
    • ATPT (Analysis Time Point): 解析時点のテキスト記述。
    • ATPTN (Analysis Time Point Number): 解析時点の数値順序。

ADaM BDSにおけるPKパラメータの構造例: | 変数名 | ラベル | 型 | 説明 | | :--------- | :--------------------------- | :--- | :--------------------------------------------- | | STUDYID | Study Identifier | CHAR | 試験識別子 | | USUBJID | Unique Subject Identifier | CHAR | 被験者識別子 | | ADAPT | ADaM Analysis Parameter | CHAR | PKパラメータ名 (例: 'AUCINF_OBS', 'CMAX') | | ADAPTN | ADaM Analysis Parameter Number | NUM | PKパラメータの数値順序 | | AVAL | Analysis Value | NUM | 解析値 (例: AUC, Cmaxの実際の数値) | | AVALU | Analysis Value Unit | CHAR | 解析値の単位 (例: 'ng*h/mL', 'ng/mL') | | ANLFL | Analysis Record Flag | CHAR | 解析対象レコードフラグ ('Y'/'N') | | PARAMCD | Parameter Code | CHAR | パラメータの短縮コード (ADaMIG準拠) | | PARAM | Parameter | CHAR | パラメータ名 (ADaMIG準拠) | | TRACEREF | Traceability Reference | CHAR | 元のSDTMレコードへの追跡情報 (例: PC.PCDTC) |

Step 3: SASコード例とマッピングロジック

ここでは、NCAツールで計算されたPKパラメータ（AUC, Cmax, Tmaxなど）がSDTMのPPドメインまたはカスタムドメイン（もしPPに合わない場合）として既にあるという前提で、それらをADaM BDSに変換する例を示します。

/* --- PKパラメータの元データ（SDTM-PP相当、またはNCAツール出力）を想定 --- */
/* 通常はNCAツールの出力ファイルやSDTM.PPから読み込みます */
DATA PK_PARAMS;
  LENGTH STUDYID USUBJID PPARM $20 PPORRES $20 PPSTRESN $20;
  INPUT STUDYID $ USUBJID $ PPARM $ PPORRES $ PPSTRESN $;
  DATALINES;
STUDY001 SUBJ001 AUCINF_OBS 1234.56 1234.56
STUDY001 SUBJ001 CMAX 123.45 123.45
STUDY001 SUBJ001 TMAX 2.0 2.0
STUDY001 SUBJ002 AUCINF_OBS 987.65 987.65
STUDY001 SUBJ002 CMAX 98.76 98.76
STUDY001 SUBJ002 TMAX 4.0 4.0
;
RUN;

/* --- SDTM.DM (被験者特性) ドメインを想定 --- */
DATA DM;
  LENGTH STUDYID USUBJID SEX $1;
  INPUT STUDYID $ USUBJID $ SEX $;
  DATALINES;
STUDY001 SUBJ001 M
STUDY001 SUBJ002 F
;
RUN;

/* --- PK ADaM BDSの作成 --- */
DATA ADPK;
  SET PK_PARAMS;

  /* ADaM必須変数の導出 */
  STUDYID = STUDYID;
  USUBJID = USUBJID;

  /* PPARMに基づきADAPT/ADAPTNを定義 */
  LENGTH ADAPT $30 AVALU $20;
  SELECT (PPARM);
    WHEN ('AUCINF_OBS') DO; ADAPT = 'AUC(0-INF)_OBS'; ADAPTN = 1; AVALU = 'ng*h/mL'; END;
    WHEN ('CMAX')       DO; ADAPT = 'Cmax';          ADAPTN = 2; AVALU = 'ng/mL'; END;
    WHEN ('TMAX')       DO; ADAPT = 'Tmax';          ADAPTN = 3; AVALU = 'h';     END;
    OTHER DO; ADAPT = PPARM; ADAPTN = .; AVALU = 'Unknown'; END; /* 未定義パラメータの場合 */
  END;

  /* 解析値 (AVAL) は数値型に変換 */
  AVAL = INPUT(PPSTRESN, BEST.);

  /* 解析対象フラグの設定 */
  ANLFL = 'Y'; /* 全てのPKパラメータを解析対象とする例 */
  PKFL = 'Y';  /* PK解析フラグも'Y'に設定 */

  /* その他のADaM必須変数 */
  PARAMCD = PPARM; /* ADaMIGで定義されるPARAMCDを使用することが推奨 */
  PARAM = ADAPT;   /* PARAMもADaMIGで定義されるPARAMを使用することが推奨 */

  /* 必要に応じて、SDTM.DMなどから被験者特性をマージ */
  /* MERGE DM BY STUDYID USUBJID; */
  /* -> この例ではシンプルにするため、マージは省略 */

  /* 変数属性 (ラベル、フォーマット) の設定 */
  LABEL ADAPT='Analysis Parameter Description'
        ADAPTN='Analysis Parameter Number'
        AVAL='Analysis Value'
        AVALU='Analysis Value Unit'
        ANLFL='Analysis Record Flag'
        PKFL='PK Analysis Flag'
        PARAMCD='Parameter Code'
        PARAM='Parameter';

  FORMAT AVAL 8.2; /* AVALの表示形式 */
RUN;

PROC PRINT DATA=ADPK;
  TITLE 'PK ADaM BDS Dataset';
RUN;

マッピングロジックのポイント: 1. トレーサビリティ (Traceability): ADaMデータセットは、SDTMデータセットや元データ（NCA出力など）との紐付けが明確である必要があります。Define-XMLでその関係性を詳細に記述します。 2. 一意性: 各被験者、各パラメータ、各解析対象において、ADaMレコードが一意になるように設計します。 3. 解析準備: 統計解析プログラム（例：SAS PROCs, R）がそのまま読み込んで使用できる形式を目指します。例えば、ADAPT変数でパラメータを識別し、AVAL変数にその値を持たせるなど、汎用的な構造が重要です。

海外の最新動向：FDA/EMAの要件とCDISC標準の更新

FDA (米国食品医薬品局) およびEMA (欧州医薬品庁) は、臨床試験データ提出におけるCDISC標準の採用を強く推奨し、義務化を進めています。特にPK解析データについては、その重要性から厳格な対応が求められています。

FDAの動向

FDAは「Study Data Technical Conformance Guide (TCG)」を定期的に更新しており、データ提出における技術的な要件を明確にしています。 * CDISC SDTM/ADaMへの準拠: PKデータを含む全ての試験データは、SDTMIG (SDTM Implementation Guide) およびADaMIG (ADaM Implementation Guide) に従って作成される必要があります。 * Define-XMLの必須化: データセットのメタデータ（変数定義、導出ロジックなど）を記述したDefine-XML 2.0または2.1の提出が必須です。特にADaMの導出ロジックは、Define-XMLの<Method>エレメントや<Source>エレメントを用いて詳細に記載することが求められます。 * PKS/PKPPドメイン: SDTMIG v3.3以降で導入されたPKS (Pharmacokinetics Subject) ドメインとPKPP (Pharmacokinetics Parameters) ドメインは、PK関連情報の標準化を促進します。NCAツールからの出力は、これらのドメインにマッピングされることが推奨されます。ただし、解析に用いる最終的なPKパラメータはADaMで作成します。

出典: * FDA Study Data Technical Conformance Guide * CDISC SDTMIG (Pharmacokinetics domains)

EMAの動向

EMAもまた、CTD (Common Technical Document) 形式でのデータ提出を求めており、CDISC標準への準拠は不可欠です。 * eSubmission Roadmap: EMAは電子提出のロードマップを公開しており、CDISC標準に基づく臨床データ提出が将来的に義務化されることを示唆しています。 * ADaMの重要性: EMAもFDAと同様に、統計解析の再現性と透明性を高めるために、ADaMの設計とDefine-XMLでの詳細な記述を重視しています。

出典: * EMA eSubmission

CDISC標準の更新情報

CDISCは、新たなユースケースや解析要件に対応するため、標準を継続的に更新しています。 * ADaMIGの更新: 最新版のADaMIGでは、より複雑な解析デザインや派生変数の定義についてガイダンスが追加されています。PK解析における欠損値の扱い方や、複数の用量群・投与経路におけるパラメータの取り扱いなど、細かな点にも注意が必要です。 * ADaM-PK Supplement: 非コンパートメント解析 (NCA) で算出されたPKパラメータをADaM形式で表現するための具体的なガイダンスを提供しています。これにより、NCA結果からADaMへのマッピングの標準化が進められています。

出典: * CDISC ADaMIG * CDISC ADaM-PK Supplement

日本での対応ポイント：PMDA固有の要件と国内実務

PMDA (医薬品医療機器総合機構) も、FDA/EMAと協調し、臨床試験データ提出におけるCDISC標準の利用を強く推奨しています。2020年以降は、原則としてSDTMおよびADaM形式でのデータ提出が求められています。

PMDAの要求事項

  • 「電磁的記録・電磁的媒体による申請に関する留意事項について」: この通知により、CDISC標準でのデータ提出が実質的に義務化されています。SDTM、ADaM、Define-XMLの各ファイルの整合性と品質が重視されます。
  • データ品質とトレーサビリティ: PMDAはデータのトレーサビリティを非常に重視します。特にPK ADaMにおいては、どのSDTMデータから、どのようなロジックでPKパラメータが導出されたのかをDefine-XMLで明確に説明する必要があります。
  • Define-XMLの重要性: Define-XMLは、PMDAの審査官が提出されたデータを理解し、確認するための「地図」です。ADaMの派生変数一つ一つについて、その計算ロジック、元のSDTM変数、使用したアルゴリズムなどを詳細に記述することが求められます。SASコードとDefine-XMLの内容が一致していることが非常に重要です。

FDAとPMDAの要件の違い

基本的なCDISC標準への準拠という点では両者に大きな違いはありませんが、PMDAは文書や説明の「きめ細かさ」を求める傾向があります。

項目 FDA要件 (傾向) PMDA要件 (傾向)
標準準拠 CDISC標準への技術的準拠を重視 CDISC標準への技術的準拠に加え、明確な説明資料を重視
Define-XML 論理的なロジック記述を重視 論理的ロジックに加え、具体的なSASコードや計算式を併記する説明を好む
トレーサビリティ メタデータで原則説明 メタデータに加え、必要に応じて別途説明資料を添付することも推奨
PK解析 ADaM-PK Supplementなどの活用を期待 ADaM-PKに加え、日本の実務に合わせた説明が求められることも

国内の実務慣行とアドバイス

  • 早期からの連携: CRO (医薬品開発業務受託機関) や製薬企業のPK担当者、データサイエンティスト、統計家は、試験計画の段階からPK ADaMの設計について密接に連携することが成功の鍵です。どのようなPKパラメータが必要か、どのような解析が予定されているかを早期に合意形成することが重要です。
  • NCAツールの活用と出力の標準化: Phoenix WinNonlinなどのNCAツールから出力されるPKパラメータデータを、直接または中間データとしてSDTM-PPやADaMに効率的にマッピングするワークフローを確立しましょう。ツールの出力形式を標準化することで、SASプログラミングの効率が向上します。
  • Define-XML作成ツールの活用: Define-XMLの手動作成は煩雑でエラーの元となりやすいため、Pinnacle 21 EnterpriseやSAS Clinical Standards Toolkitなどのツールを活用して、自動生成・検証を行うことを強く推奨します。これにより、品質と効率を両立できます。
  • バリデーションの徹底: 作成されたADaMデータセットは、Pinnacle 21 Validatorなどのツールを用いて徹底的にバリデーションを行います。エラーや警告を早期に発見し、修正することで、最終的な申請時の手戻りを防ぎます。

よくあるエラーと対処法：バリデーションの落とし穴

ADaMデータセットの作成は、細かなルールが多く、Pinnacle 21などのバリデーションツールで多くのエラーや警告が発生しがちです。特にPK ADaMにおいてよく見られるエラーとその対処法をいくつか紹介します。

1. ADaM変数属性の不一致

  • エラー内容: ADaMIG (ADaM Implementation Guide) で定義されている変数名、ラベル、タイプ（文字/数値）、長さ、フォーマットと、実際のデータセットで定義されているものが一致しない。
  • 発生要因: コピー＆ペーストミス、手動での変数定義の誤り、ADaMIGの古いバージョンを参照しているなど。
  • 対処法:
    • 必ず最新のADaMIGを参照し、変数定義テーブル (VADAM) を確認します。
    • SASプログラムで変数属性を定義する際は、ATTRIBステートメントやLENGTHステートメントを正確に使用します。
    • ADaMテンプレートライブラリを作成し、標準的な変数はそこから利用することで一貫性を保ちます。

2. Define-XMLとの不整合

  • エラー内容: Define-XMLで記述されている変数の導出ロジックや属性が、実際のADaMデータセットやSASプログラムと一致しない。
  • 発生要因: ADaMデータセットの作成後にロジックが変更されたが、Define-XMLが更新されていない。Define-XMLの記述自体に誤りがある。
  • 対処法:
    • ADaMデータセットの最終版が確定した後にDefine-XMLを作成・更新します。
    • 可能であれば、SASプログラムからDefine-XMLの一部を自動生成する仕組みを導入します。
    • Pinnacle 21 ValidatorはDefine-XMLとデータセットの整合性チェックも行いますので、必ずP21で確認し、エラーを解消します。

3. PKパラメータ値の不適切性

  • エラー内容: AUCが負の値になる、CmaxがPCORRESの最大値と一致しない、Tmaxが測定時点の範囲外であるなど、PKパラメータの値に論理的な矛盾がある。
  • 発生要因: 元のPCデータに問題がある（欠損値、異常値）、NCAツールの計算設定ミス、SASでの値の取り込みや変換ミス。
  • 対処法:
    • SDTM.PCデータの品質を十分に確認し、データクリーニングを徹底します。
    • NCAツールでの計算ロジックや設定を厳密にレビューします。
    • SASでADaMを作成する際に、PKパラメータの妥当性チェック（例：AVALが正の値か、TMAXがPCデータ範囲内かなど）を組み込むことも有効です。

4. 欠損値の扱いの不統一

  • エラー内容: 欠損値の表現 (SASの.、.Aなど) や、その欠損理由の記述 (ADaM変数 AVALC、DTYPEなど) がADaMIGや試験計画書と異なる。
  • 発生要因: 欠損値の取り扱いや表現に関する合意形成が不足している、プログラマーによって実装が異なる。
  • 対処法:
    • 欠損値の取り扱いに関する共通のルール（標準作業手順書: SOP）を確立します。
    • ADaMIGに沿って、欠損値の種類に応じて適切な DTYPE や AVALC を使用します。
    • 例えば、測定下限値 (LLOQ) 未満のデータは、通常、SDTMではPCORRESを欠損とし、PCSTAT='NOT DONE'、PCREAS='BELOW QUANTIFICATION'などと記載します。ADaMでこれらをどのように扱うか（例: 0に置き換える、特定の欠損コードにする）は、統計解析計画書 (SAP) に明記します。

よくある質問（FAQ）

Q1: PK解析ADaMは必ず必要ですか？

A1: はい、多くの規制当局（FDA、PMDAなど）は、統計解析計画書 (SAP) に基づく主要な解析結果を導出するために使用された解析データセット (ADaM) の提出を義務付けています。PK解析も主要な解析の一つであるため、PKパラメータを導出したADaMデータセットは原則として必要となります。これにより、解析の再現性と透明性が確保されます。

Q2: ADaMのトレーサビリティはどのように担保しますか？

A2: ADaMのトレーサビリティは、主にDefine-XMLファイルを通じて担保されます。Define-XMLには、ADaMの各変数がどのSDTM変数から、どのようなロジック（計算式や条件）で導出されたかを詳細に記述します。また、SASコード内でコメントを活用し、各処理ステップと元のデータとの関連性を明確にすることも重要です。適切な変数ラベルやコメントをSASコードとDefine-XMLの両方で維持することで、審査官が容易にデータの流れを追跡できるようになります。

Q3: SDTMのPCドメインとADaMのBDSはどう関係しますか？

A3: SDTMのPCドメインは、被験者から採血された時点ごとの薬物濃度測定値とその生データレベルの情報を標準形式で保持します。一方、ADaMのBDS (Basic Data Structure) は、そのPCドメインのデータ、またはNCAツールによって算出されたPKパラメータを、統計解析に直接使用できる形式で整理したものです。具体的には、PCドメインの時系列の濃度データを用いて、CmaxやAUC、TmaxといったPKパラメータがNCAツールで計算され、その結果がADaMのBDSとして、各被験者・各パラメータにつき1レコードの形式で格納されるのが一般的です。ADaMは、SDTMから派生した「解析のための要約データ」と考えると分かりやすいでしょう。
