結論から言うと

臨床データの承認申請には、FDAおよびPMDAともにCDISCデータ標準に準拠した電子提出が必須です。特に、最新のStudy Data Technical Conformance Guide (SDTCG) や各当局の電子提出ガイドラインを熟読し、Pinnacle 21などのバリデーションツールを開発の早期段階から活用することが、スムーズな規制提出と承認取得の鍵となります。常に海外の最新動向を把握し、日本の実務に落とし込む視点が成功には不可欠です。

臨床データの承認申請・規制提出とは？

医薬品や医療機器が承認され、患者さんのもとに届けられるためには、その安全性と有効性を示す臨床データを、各国の規制当局（日本ではPMDA、米国ではFDA）に提出し、審査を受ける必要があります。かつては紙媒体での提出が主流でしたが、IT技術の進歩に伴い、現在では電子データでの提出が世界的に義務化、または強く推奨されています。

この電子データ提出において中心的な役割を果たすのが、CDISCデータ標準です。CDISCは、臨床データの収集、交換、提出、アーカイブの方法に関する国際的な標準を開発する非営利団体です。CDISC標準を用いることで、データの一貫性が保たれ、規制当局の審査効率が向上し、結果として新薬がより早く患者さんに届くことに貢献します。

主要なCDISC標準には以下のものがあります。

  • SDTM (Study Data Tabulation Model): 症例報告書 (CRF) や電子データキャプチャ (EDC) で収集された個々の被験者レベルの生データ（あるいはそれに近い詳細なデータ）を標準化するためのモデルです。例えば、被験者の背景情報、有害事象、併用薬、検査値などがSDTMドメインとして構造化されます。
  • ADaM (Analysis Data Model): 統計解析を行うためのデータセットのモデルです。SDTMデータから統計解析に必要な変数を抽出し、解析可能な形式に加工することで、解析の再現性と透明性を高めます。
  • Define-XML: SDTMやADaMデータセットの構造、変数、ロジック、Controlled Terminology（管理用語）などを機械可読なXML形式で記述したメタデータです。規制当局が提出されたデータを理解するために不可欠な「データの辞書」のようなものです。
  • CT (Controlled Terminology): CDISCが管理する標準的な用語集です。性別、投与経路、疾患名、評価項目など、多くの項目において特定の用語を使用することで、データの一貫性と解釈の正確性を保証します。

これらの標準に従って作成されたデータセットは、通常、eCTD（electronic Common Technical Document）という国際的な電子申請フォーマットの一部として規制当局に提出されます。eCTDは、申請資料全体を構造化し、電子的に提出するための枠組みであり、臨床データはその中で「Module 4（非臨床試験の概要）」「Module 5（臨床試験の概要）」として位置づけられます。

実務での使い方・実装手順

実際に臨床データをPMDAやFDAへ規制提出する際の、SDTMやADaM、Define-XMLの作成、およびバリデーションのステップは以下のようになります。

  1. SDTMデータセットの作成: 収集された生データ（例：EDCデータ）をSDTMIG (Implementation Guide) に従ってSDTMドメインにマッピングします。この工程が臨床データ標準化の基盤となります。
  2. ADaMデータセットの作成: SDTMデータから、統計解析計画書 (SAP) に基づいて解析に特化したADaMデータセットを作成します。これには、統計解析に必要な変数の導出や、解析に合わせたデータ構造の変更などが含まれます。
  3. Define-XMLの作成: 作成したSDTM、ADaMデータセットの構造、変数属性、導出ロジック、使用したCTなどをDefine-XMLとして記述します。これは手動で作成することも可能ですが、通常は専用ツールやSASプログラムで自動生成されます。
  4. Controlled Terminology（CT）の適用: SDTMやADaMデータセットで使用される全てのコードや用語が、最新のCDISC CTに準拠しているか確認し、必要に応じてマッピングまたは変換します。
  5. バリデーションの実施: Pinnacle 21などの専用ツールを用いて、作成したSDTM、ADaM、Define-XMLデータセットがCDISC標準および規制当局の技術的要件に準拠しているかを確認します。このステップは早期から繰り返し実施することが非常に重要です。
  6. eCTDへのパッケージング: CDISCデータセットおよび関連するドキュメント（CRF、プロトコル、統計解析計画書など）をeCTDの指定されたディレクトリ構造に配置し、申請パッケージを完成させます。

SASコード例：SDTM.AE（有害事象）ドメインの作成

ここでは、一般的な有害事象の生データから、SDTMのAEドメインの一部を作成する簡単なSASコード例をご紹介します。実際のプロジェクトではより複雑なロジックが必要となりますが、基本的な考え方は同じです。

```sas data raw_ae; infile datalines dlm=','; input SUBJID $ AETERM $ AESTDAT date9. AESER $ AE_CAUSAL_REL $; format AESTDAT yymmdd10.; datalines; 001,Headache,2023-01-10,N,POSSIBLE 002,Nausea,2023-01-15,Y,PROBABLE 001,Fever,2023-01-12,Y,DEFINITE 003,Dizziness,2023-01-20,N,NOT RELATED ;

data sdtm.ae; set raw_ae; / SDTM Required Variables / STUDYID = 'STUDY001'; / 試験ID / DOMAIN = 'AE'; / ドメイン名 / USUBJID = catx('-', STUDYID, SUBJID); / ユニーク被験者ID / AETERM = AETTER; / 報告された有害事象名 / AESTDTC = put(AESTDAT, yymmddn8.); / 有害事象開始日時 (ISO 8601形式) / AESER = upcase(strip(AESER)); / 重篤性 'Y' or 'N' / / Conditional Variables & Mapping / if AESER = 'Y' then AESER = 'Y'; / Y/Nをそのまま使用 / else if AESER = 'N' then AESER = 'N'; else AESER = ''; / 不明な場合は空白 /

/ Additional Mapping for Causal Relationship (Example) / if upcase(strip(AE_CAUSAL_REL)) = 'DEFINITE' then AECRLT = 'DEFINITELY RELATED'; else if upcase(strip(AE_CAUSAL_REL)) = 'PROBABLE' then AECRLT = 'PROBABLY RELATED'; else if upcase(strip(AE_CAUSAL_REL)) = 'POSSIBLE' then AECRLT = 'POSSIBLY RELATED'; else if upcase(strip(AE_CAUSAL_REL)) = 'NOT RELATED' then AECRLT = 'NOT RELATED'; else AECRLT = ''; / その他の場合は空白 / run;
